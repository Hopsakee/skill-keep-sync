# Preparing a Branch for Pull Request

Stages all changes on the current branch, generates a comprehensive PR review markdown document explaining every change, commits, pushes, and tells the user how to open the PR on GitHub.

## Workflow

### Phase 1: Gather Context

1. Get the current branch name:
   ```bash
   git branch --show-current
   ```
2. Identify the base branch (usually `main`). If unsure, check:
   ```bash
   git log --oneline main..HEAD
   ```
   If `main` doesn't work, try `master` or ask the user.

3. Get the full diff between the base branch and the **working tree** (includes both committed and uncommitted changes):
   ```bash
   git diff --stat main
   git diff main
   ```
   For large diffs, break it into logical groups:
   ```bash
   git diff main -- src/module_a.py src/module_b.py
   git diff main -- tests/
   ```

4. Check for untracked files that should be included:
   ```bash
   git status --short
   ```

5. Read the commit history on this branch for context:
   ```bash
   git log --oneline main..HEAD
   ```

### Phase 2: Analyze and Categorize Changes

Group changes into logical categories. Common groupings:
- **Infrastructure / build changes** (pyproject.toml, CI, config files)
- **Data model changes** (database schemas, dataclasses, models)
- **Module rewrites / migrations** (API changes, framework migrations)
- **Bug fixes** (broken imports, logic errors)
- **New files** (new modules, documentation, config)
- **Documentation updates** (README, docs, comments)
- **Test changes** (new tests, fixed tests, disabled tests)
- **Cleanup** (removed files, dead code removal)

For each change, understand:
- **What** changed (the code diff)
- **Why** it changed (the motivation — fix a bug, migrate an API, add a feature)
- **Impact** (what other code is affected)

### Phase 3: Write the PR Review Document

Create a markdown file at `docs_md/pr/<branch-name>.md`.

Create the `docs_md/pr/` directory if it doesn't exist.

#### Document Structure

```markdown
# PR: <Short descriptive title>

**Branch:** `<branch-name>`
**Date:** <today's date>

## Summary

Brief overview of all changes (bulleted list, one line per logical change).

---

## 1. <Category Name>

### <Specific Change>

Explain what changed and why.

**Before:**
\```python
<old code>
\```

**After:**
\```python
<new code>
\```

**Why:** <motivation for the change>

---

## 2. <Next Category>

...
```

#### Writing Guidelines

- **Use ample code blocks** — show before/after for every significant change
- **Explain the "why"** — not just what changed, but the motivation
- **Group related changes** — don't list every file separately if they share the same change (e.g., "all modules updated import from X to Y")
- **Number the top-level sections** for easy reference during review
- **Call out renamed functions/APIs** explicitly — reviewers need to know what moved
- **Highlight breaking changes** or behavior differences
- **Note any remaining TODOs or follow-up work** if applicable
- **Keep code blocks focused** — show the relevant function/block, not entire files
- For trivial/mechanical changes (e.g., auto-generated cell IDs, execution counts in notebooks), summarize rather than listing each one

### Phase 4: Stage All Changes

```bash
git add -A
```

Verify what's staged:
```bash
git status --short
```

### Phase 5: Commit

Write a descriptive commit message with:
- A concise subject line (imperative mood, under 72 chars)
- A blank line
- Bullet points summarizing the key changes

```bash
git commit -m "Subject line here

- First key change
- Second key change
- Third key change"
```

### Phase 6: Push

```bash
git push --set-upstream origin <branch-name>
```

If the branch already tracks a remote, just:
```bash
git push
```

### Phase 7: Tell the User How to Create the PR

After pushing, provide the user with:

1. The direct GitHub URL to create the PR:
   ```
   https://github.com/<owner>/<repo>/compare/<base>...<branch>
   ```
   Extract owner/repo from:
   ```bash
   git remote get-url origin
   ```

2. Instructions to paste the contents of `docs_md/pr/<branch-name>.md` as the PR body.

Example output:
> ✅ All changes staged, committed, and pushed.
>
> To create the PR, go to:
> 👉 **https://github.com/owner/repo/compare/main...branch-name**
>
> Paste the contents of `docs_md/pr/branch-name.md` as the PR description.

## Tips

- If the diff is very large (50+ files), prioritize the most impactful changes and summarize mechanical/generated changes briefly
- For nbdev projects: notebook diffs are noisy (execution counts, output cells). Focus on the source cell changes, not the metadata
- For auto-generated files (e.g., `_modidx.py`, compiled outputs), just mention they were regenerated — don't show diffs
- If log files or binary files are in the diff, mention them but don't detail their contents
