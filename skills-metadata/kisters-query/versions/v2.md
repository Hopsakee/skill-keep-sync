## MANDATORY: Discover before fetching

**Always run discovery first** ŌĆö before fetching any measurements, run `--discover` to see every site, station, and timeseries that matches the location name, including actual data coverage dates (`from` / `to`). Show the output to the user and ask them to confirm (or pick) the correct site/station if the result is ambiguous or has multiple options.

Only proceed to fetch data once you know exactly which `ts_id` (preferred) or `site_no` + `station_no` + `ts_shortname` to use.

```bash
# Step 1 ŌĆö discover (always do this first)
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "Katerveer" --parameter H --discover

# Step 2 ŌĆö fetch using ts_id from discovery (preferred: avoids station resolution errors)
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "Katerveer" --parameter H \
    --ts-id 1084858042 \
    --from 2024-09-01 --to 2024-10-31

# Step 2 (alternative) ŌĆö fetch with explicit site/station/shortname
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "Katerveer" --parameter H \
    --site-no WDOD_17679 --station-no MPN_WDOD_17679_P010 --ts-shortname mean.1d \
    --from 2024-09-01 --to 2024-10-31
```

### When the location name doesn't match KiWIS site names

`fetch_timeseries.py --discover` searches KiWIS site names with wildcards. If the user's location name doesn't match any KiWIS site name (e.g. "Balgstuw" ŌĆö the KiWIS site is called "10471S Inlaat Panjerd Veeningen"), use this two-step pipeline:

```bash
# Step 1 ŌĆö find the site geographically
uv run python .windsurf/workflows/kisters-query/scripts/find_by_geography.py \
    --near "Balgstuw" --parameter H --from 2025-12-01 --to 2025-12-31 --confirm

# Step 2 ŌĆö discover all stations and timeseries at the found site
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --site-no WRW_7002 --parameter H --discover
```

**Always run step 2** after `find_by_geography.py` ŌĆö a site can have multiple stations (e.g. Hoogwaterzijde and Laagwaterzijde at a weir) and you need to see the full station structure before fetching data.

## Multi-station queries (bovenstrooms / benedenstrooms)

Hydraulic structures (stuwen, gemalen, inlaten) often have **multiple stations** at one site:

| Station name | Meaning |
|---|---|
| **Hoogwaterzijde** | Upstream (bovenstrooms) |
| **Laagwaterzijde** | Downstream (benedenstrooms) |
| **Stuw** | Weir/dam itself |
| **Klep** | Gate/valve |

When the user asks for measurements at multiple points (e.g. "bovenstrooms ├®n benedenstrooms"), **always use `--discover` first** to see all stations. Then fetch each timeseries separately using `--ts-id`:

```bash
# 1. Discover all stations and timeseries at the site
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --site-no WRW_7002 --parameter H --discover

# 2. Fetch upstream (Hoogwaterzijde) ŌĆö use ts_id from discovery output
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --ts-id 1085802042 --from 2025-12-01 --to 2025-12-31 \
    --output balgstuw_hoogwaterzijde_dec2025.csv

# 3. Fetch downstream (Laagwaterzijde) ŌĆö use ts_id from discovery output
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --ts-id 1085822042 --from 2025-12-01 --to 2025-12-31 \
    --output balgstuw_laagwaterzijde_dec2025.csv
```

> **Tip:** Prefer `UurGem` timeseries for hourly data, `DagGem` for daily averages, and `Momentaan` for raw high-frequency measurements.

### `find_by_geography.py` ŌĆö find all sites near a geographic location or waterway

Resolves the location via OpenStreetMap (Nominatim). For waterways and areas (e.g. Meppelerdiep) the OSM bounding box is used; for point locations (e.g. a city) a radius around the centroid is used. Filters by parameter type using a two-phase approach that correctly handles compound parameter names.

```bash
# All water level sites near the Meppelerdiep (default 1 km buffer for waterways)
uv run python .windsurf/workflows/kisters-query/scripts/find_by_geography.py \
    --near "Meppelerdiep" --parameter H

# All discharge sites within 7 km of Zwolle that have 2024 measurements
uv run python .windsurf/workflows/kisters-query/scripts/find_by_geography.py \
    --near "Zwolle" --parameter Q --radius 7 --from 2024-01-01 --to 2024-12-31

# All sites with any parameter, near the Meppelerdiep, measured in 2024
uv run python .windsurf/workflows/kisters-query/scripts/find_by_geography.py \
    --near "Meppelerdiep" --from 2024-01-01 --to 2024-12-31
```

**Arguments:**

| Argument | Required | Description |
| --- | --- | --- |
| `--near` | yes | Location name ŌĆö waterway, city, area (e.g. `"Meppelerdiep"`, `"Zwolle"`) |
| `--radius` | no | Override radius/buffer in km. **Default: 1 km for waterways/areas, 5 km for cities/points.** |
| `--parameter` | no | Parameter type: `Q`, `H`, `V`, `P`. Omit to return all. |
| `--from` | no | Only include sites with data from this date `YYYY-MM-DD` |
| `--to` | no | Only include sites with data up to this date `YYYY-MM-DD` |
| `--output` | no | CSV filename in `data/` (auto-generated if omitted) |
| `--confirm` | no | Skip the >25-site warning and proceed with Phase 2 anyway |

**How it determines the search area:**

- Looks up the location via Nominatim (OpenStreetMap).
- If the OSM feature has a meaningful bounding box (> 0.5 km, e.g. a waterway or polder): the OSM bbox + **1 km buffer** is used by default.
- If the OSM result is a point (city, address): a circle of **5 km radius** around the centroid is used by default.
- Override either default with `--radius`.
- A haversine distance from the centroid is always computed and shown in the output.

**How it filters when `--parameter` is given (two-phase):**

- **Phase 1 (fast):** `getParameterList` per site ŌĆö filters sites that have the parameter type. Handles compound names like `Q [m3/s] [NVT] [OW]` that the server-side filter misses. Always runs without a warning, even for large areas.
- **Phase 2 (slower):** `getTimeseriesList` with `coverage` per station ŌĆö verifies that the matching timeseries actually contains measurements (excludes empty timeseries). Also checks date overlap if `--from`/`--to` are given. Warns if >25 sites remain after Phase 1 (use `--confirm` to proceed).

### `fetch_timeseries.py` ŌĆö fetch measurements for a location + parameter

The workhorse script. Handles site/station discovery, parameter filtering, timeseries resolution selection, and saves results to `data/`.

```bash
# Discharge at a location for a date range
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "gemaal Stroink" --parameter Q \
    --from 2024-12-01 --to 2024-12-31

# Water level at a location for the last 30 days
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "Stroink" --parameter H --period P30D

# Custom output filename
uv run python .windsurf/workflows/kisters-query/scripts/fetch_timeseries.py \
    --location "Stroink" --parameter Q \
    --from 2024-01-01 --to 2024-12-31 --output stroink_q_2024.csv
```

**Arguments:**

| Argument | Required | Description |
| --- | --- | --- |
| `--location` | conditional | Location name or partial name (e.g. `"Stroink"`, `"gemaal Stroink"`). Not required when `--site-no` or `--ts-id` is given. |
| `--parameter` | conditional | Parameter type: `Q`, `H`, `V`, `P`. Not required when `--ts-id` is given. Required for `--discover`. |
| `--from` | no | Start date `YYYY-MM-DD` |
| `--to` | no | End date `YYYY-MM-DD` |
| `--period` | no | ISO 8601 period, e.g. `P30D`, `P1Y` (alternative to `--from`/`--to`) |
| `--output` | no | CSV filename in `data/` (auto-generated if omitted) |
| `--discover` | ŌĆö | List all matching sites/stations/timeseries with coverage dates, then exit. **Run this first.** |
| `--site-no` | no | Override automatic site selection (use exact `site_no` from discovery) |
| `--station-no` | no | Override automatic station selection (use exact `station_no` from discovery) |
| `--ts-shortname` | no | Override timeseries selection by shortname (e.g. `momentaan.all`, `cmd.CS.p`, `mean.1h`, `mean.1d`) |
| `--resolution` | no | Preferred resolution: `raw` (default), `hourly`, `daily` |
| `--ts-id` | no | **Skip all resolution** ŌĆö fetch directly with this `ts_id` from discovery. Prefer this over `--ts-shortname` when the ts_id is known. |

**How it selects the right site/station (when not overridden):**

- Searches `getSiteList` with a wildcard on the location name; if "gemaal Stroink" yields no match it falls back to "Stroink" automatically.
- Picks the site whose `site_type_name` best matches the parameter (e.g. "Gemaal" for `Q`).
- Within that site, picks the station whose name best matches (e.g. "Gemaal" station for `Q`).
- Default resolution is `raw`: prefers `momentaan.all` ŌåÆ `momentaanall` ŌåÆ `momentaan.v` ŌåÆ `cmd.CS.p` ŌåÆ `mean.1h` ŌåÆ `mean.1d`.
- Use `--resolution hourly` to prefer `mean.1h`, `--resolution daily` to prefer `mean.1d`.

> **Prefer `--ts-id` over `--ts-shortname`** when you know the exact timeseries from discovery ŌĆö it bypasses all station resolution and is immune to station_no lookup issues.
>
> **Always prefer `--site-no` / `--station-no` / `--ts-shortname` over relying on auto-selection**, especially when discovery reveals multiple matching sites.

## Data Hierarchy

```text
Site  ŌåÆ  Station  ŌåÆ  StationParameter  ŌåÆ  Timeseries  ŌåÆ  TimeseriesValues
         (physical     (what is            (one          (actual
          location)     measured)           aggregation)   readings)
```

- A **Site** groups one or more Stations (e.g. a pumping station area, catchment).
- A **Station** is the physical sensor/measurement point; belongs to exactly one Site.
- A **StationParameter** links a Station to a ParameterType (e.g. Station X measures H).
- A **Timeseries** is one specific aggregation/resolution per StationParameter (e.g. `totals.1d`).
- **Timeseries path** format: `site_no/station_no/stationparameter_no/ts_shortname`

### Discharge parameters (`Q`, m┬│/s)

Discharge in Dutch water management uses parameter type **`Q`**.
The server-side `parametertype_name` filter may not match compound names like `Q [m3/s] [NVT] [OW]`,

```python
discharge = df_params[
    df_params["stationparameter_name"].str.contains("Debiet", case=False, na=False)
    | df_params["parametertype_unitname"].str.contains("m3/s", case=False, na=False)
]
```

### Sites within X km of a city

```python
import math
from kiwis import kiwis

def get_city_coords(city: str) -> tuple[float, float]:
    """Return (lat, lon) for a city name via Nominatim."""
    import requests
    r = requests.get(
        "https://nominatim.openstreetmap.org/search",
        params={'q': city, 'format': 'json', 'limit': 1},
        headers={'User-Agent': 'wiski-llm-test/1.0'},
        timeout=10,
    )
    r.raise_for_status()
    results = r.json()
    if not results:
        raise ValueError(f"City not found: {city}")
    return float(results[0]["lat"]), float(results[0]["lon"])


def haversine_km(lat1: float, lon1: float, lat2: float, lon2: float) -> float:
    """Return great-circle distance in km between two WGS84 points."""
    R = 6371.0
    dlat = math.radians(lat2 - lat1)
    dlon = math.radians(lon2 - lon1)
    a = math.sin(dlat / 2) ** 2 + (
        math.cos(math.radians(lat1)) * math.cos(math.radians(lat2)) * math.sin(dlon / 2) ** 2
    )
    return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1 - a))


df_sites = kiwis(
    "getSiteList",
    returnfields="site_id,site_no,site_name,site_latitude,site_longitude,site_type_name",
)
df_sites = df_sites[(df_sites["site_latitude"] != "") & (df_sites["site_longitude"] != "")].copy()
df_sites["site_latitude"] = df_sites["site_latitude"].astype(float)
df_sites["site_longitude"] = df_sites["site_longitude"].astype(float)

lat, lon = get_city_coords("Zwolle, Netherlands")
df_sites["distance_km"] = df_sites.apply(
    lambda row: haversine_km(lat, lon, row["site_latitude"], row["site_longitude"]), axis=1
)
nearby = df_sites[df_sites["distance_km"] <= 10].sort_values("distance_km")
print(nearby[["site_no", "site_name", "site_type_name", "distance_km"]])
```

## Tips & Gotchas

- **Zwolle coordinates:** ~52.5168┬░N, 6.0830┬░E ŌĆö not in the database but can be resolved via Nominatim.
- **Coordinates can be empty strings** in the JSON ŌĆö always filter before converting to float.
- **`coverage` returnfield** on `getTimeseriesList` adds `from`/`to` fields showing actual data range.
- **`ts_path`** uniquely identifies a timeseries: `site_no/station_no/stationparameter_no/ts_shortname`
- **`flatten=true`** on `getStationList` collapses multi-parameter rows into one row per station.
- **Wildcard `*`** works on fields marked "yes" for wildcard (e.g. `site_name=03*` for all names starting with 03).
- **Comma-separated lists** work on fields marked "yes" for comma list (e.g. `site_no=WRW_03_02,WRW_03_03`).
- **`bbox`** uses WGS84 lon/lat by default. Add `crs=local` to use local cartesian coordinates (RD New).
- **`parametertype_name` server filter** may not match compound names like `Q [m3/s] [NVT] [OW]` ŌĆö filter client-side with `str.contains` on `stationparameter_name` or `parametertype_unitname` instead.
- **`getTimeseriesList` + `site_no` ŌåÆ 500 error:** this KiWIS instance does not support `site_no` on `getTimeseriesList`. Always use `station_no` instead (first fetch stations via `getStationList`).
- **Station names at hydraulic structures:** Hoogwaterzijde = upstream (bovenstrooms), Laagwaterzijde = downstream (benedenstrooms), Stuw = weir, Klep = gate.
